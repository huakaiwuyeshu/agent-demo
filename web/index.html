记忆系统：为什么不让 LLM 重复问
🎯
核心价值 自动记住用户提供的信息，跨轮合并字段 → 用户说过一次 appid 就不用再问，避免重复收集
🧠
结构化记忆 vs Prompt 拼接 代码存储字段（appid、error_code 等）并自动合并，比把历史对话全塞给 LLM 更精准且省 token
📚
短期 + 长期双层记忆 短期：Session 内累积字段（刷新可恢复）| 长期：TF-IDF 检索历史案例（类似问题自动参考解决方案）
⚡
增量补全 第 1 轮给 appid → 第 2 轮给 error_code → 自动合并 → 字段齐全后直接执行，无需用户重复提供
💾 短期记忆：20 轮对话 + 字段累积（24h localStorage）| 📚 长期记忆：100 个历史任务（TF-IDF 检索）
Schema 校验：为什么不让 LLM 猜
🎯
核心价值 用代码规则保证字段齐全，而不是让 LLM 猜测或事后补救 → 一次性收集完整信息，避免多轮试错
⚖️
规则驱动 vs LLM 判断 代码检查"缺少 X、Y、Z"（快速、精确、零成本）比 LLM 判断"问题太模糊"（慢、主观、烧钱）更可靠
🔍
精准追问 不问"能否更明确？"而问"还缺少：接口路径、签名前字符串、错误码" → 用户一次补全
🛡️
执行保证 + 安全门控 字段不全 = 拒绝调用工具（防止报错）+ 敏感信息请求直接阻断（appsecret 查询拦截）
📋 签名排查 5 字段 | 📞 回调排查 4 字段 | ❓ 字段问答 1 字段 | 🚫 密钥申请触发人工闸门
Skill SOP：为什么不让 LLM 现编
🎯
核心价值 把专家经验固化为标准 SOP 文档，注入 LLM Context → 保证回答质量一致，而不是每次现场发挥
📖
SOP 文档 vs 零样本推理 15 步签名排查 SOP（参数排序 → timestamp → 签名算法）比让 LLM 凭通用知识猜测更专业可靠
🔄
持续迭代 SOP 文档可编辑、可版本管理 → 发现新问题立即更新 SOP，所有 Agent 回答自动升级（无需重新训练）
⚖️
质量保证 按 task_type 自动路由到对应 Skill → 签名问题必走签名 SOP、回调问题必走回调 SOP，避免答非所问

🎯 Agent 能力演进：从黑盒到白盒

同一个签名排查问题，不同版本如何处理？点击下方阶段，体验 Agent 的渐进式升级。

阶段 1：意图识别与字段校验

用户输入： "你们接口一直签名失败，帮忙看下"
V0 普通问答

"请提供接口路径、appid、签名前字符串、错误码和请求时间，以便帮您排查问题。"

❌ 泛泛要求 ❌ 未结构化
V1 任务识别

"识别为：signature_debug
提取字段：{}（空）
需要补充：appid、error_code..."

✅ 识别任务 ⚠ 未校验字段
V2 Schema 校验

"还缺少关键信息：
• api_path（接口路径）
• appid
• raw_sign_string
• error_code
• request_time"

✅ 精准追问 ✅ 列出必填项

💡 对比要点

  • V0 → V1：从"黑盒回答"到"识别出这是签名排查任务"
  • V1 → V2：从"知道是什么任务"到"精准指出缺少哪 5 个字段"

阶段 2：Skill 路由

前提：用户已补充所有必填字段
输入："appid=app_001, error_code=SIGN_INVALID, api_path=/open/order..."
V3 Skill 路由
🎯 路由逻辑

📖 加载 SOP

1. 检查参数是否按字典序排序

✅ 效果

Agent 回答遵循标准 SOP，不再凭空猜测
