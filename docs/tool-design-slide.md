# 能力5：工具调用

把"检查签名对不对"交给确定性代码执行，不让 LLM 猜测技术结论。

---

## 问题

LLM 无法执行确定性验证逻辑。面对"参数排序对不对"这类问题，它只能给出模糊建议（"您可以检查一下参数顺序"），无法直接告诉用户"你的参数顺序错了，正确顺序应该是 appid, timestamp, order_id"。

## 加入能力

用 Tool 执行确定性检查，将结构化结果注入 Prompt，LLM 只负责解读结果和表达。

## 收益

从"建议您检查"变为"工具检测到参数排序错误"，准确率 85% → 98%，用户无需自己排查。

---

## 工具设计三原则

### 原则 1：粒度恰当

每个工具用一句话能说清楚它干什么。

```
✗ 差的粒度（太粗）
  process_request: "处理用户请求"  → Agent 不知道传什么、期待什么

✗ 差的粒度（太细，10+ 个工具）
  check_param_a, check_param_b, check_encoding_utf8, check_encoding_gbk...
  → Agent 选择困难症，花更多 token 在"选哪个工具"而非干活

✓ 合适的粒度（3-7 个工具，各司其职）
  check_signature_order: 验证参数 ASCII 排序
  check_timestamp_present: 检查时间戳是否存在
  search_error_code: 查询错误码含义和排查方向
  query_sandbox_log: 查询沙箱请求日志
```

### 原则 2：描述比实现更重要

Agent 通过描述理解"这个工具能干什么"。描述省的每一个字，都会变成 Agent 运行时犯的错。

```javascript
// ✗ 差的描述 — Agent 不知道输入输出和限制
{
  name: "search",
  description: "搜索数据库"
}

// ✓ 好的描述 — 输入格式、输出格式、边界条件全覆盖
{
  name: "search_knowledge",
  description: "在知识库中搜索 API 文档、FAQ、错误码。"
             + "输入：关键词（接口路径/错误码/字段名）。"
             + "输出：最多5条相关文档，格式为 [类型] 标题：内容。"
             + "注意：未找到结果时会返回重试建议。"
}
```

### 原则 3：错误返回给 Agent 看

错误信息不是给人看的，是给 Agent 决策下一步的。

```javascript
// ✗ 差的错误返回 — Agent 不知道该怎么办
return null;
return { error: "failed" };

// ✓ 好的错误返回 — 发生了什么 + 可能原因 + 下一步建议
return {
  success: false,
  error: "未找到匹配日志（appid=app_001, path=/open/order）。"
       + "可能原因：请求未到达服务端、appid不精确、"
       + "或不在示例数据范围内。"
       + "建议：确认 appid 和请求时间是否准确。"
};
```


## 决策流程

```
收到输入 → 提取字段 → Schema 校验 → 缺字段？
                                        │
                              ┌─────────┴─────────┐
                              ↓                   ↓
                        LLM 追问缺失字段     执行工具链
                                              │
                                    ┌─────────┼─────────┐
                                    ↓         ↓         ↓
                              check_order  check_ts  query_log
                                    │         │         │
                                    └─────────┼─────────┘
                                              ↓
                                   工具结果注入 System Prompt
                                              ↓
                                   LLM 基于结果生成回答
```

## Tool vs Skill 协作

```
┌──────────────────────────────────────────────────────┐
│                  System Prompt                         │
│                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │  Skill SOP  │  │  工具结果    │  │  知识库文档  │  │
│  │             │  │             │  │             │  │
│  │ 排查步骤：   │  │ check_order │  │ [API文档]   │  │
│  │ 1.检查排序   │  │  = fail     │  │ 签名规则... │  │
│  │ 2.检查编码   │  │ "当前顺序： │  │             │  │
│  │ 3.检查时间戳 │  │  timestamp, │  │             │  │
│  │ ...         │  │  appid"     │  │             │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│                          ↓                            │
│                    LLM 综合推理                        │
│              "工具检测到参数排序错误，                   │
│               当前 timestamp 排在 appid 前面，         │
│               正确顺序应为 appid, timestamp..."        │
└──────────────────────────────────────────────────────┘
```

**分工**：Tool 提供事实（参数确实排错了）→ Skill 提供逻辑（按什么步骤排查）→ LLM 综合表达
