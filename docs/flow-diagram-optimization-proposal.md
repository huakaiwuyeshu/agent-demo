# Agent 交互架构图优化建议

基于已有的设计规范（Open Design、记忆系统、Schema/Skill 设计），对当前架构图的优化建议。

---

## 📊 当前架构图评估

### ✅ **做得好的地方**

1. **层次清晰**：5 列结构（用户 → 门控 → LLM → Skills → 输出）
2. **简洁紧凑**：移除了详细卡片后，阅读体验大幅提升
3. **标签化展示**：底部辅助区域用 chip 组件，信息密度合理
4. **视觉一致**：配色方案统一（蓝色 Schema、绿色 Skill）

---

### ⚠️ **存在的问题**

#### **问题 1：术语不统一**

**现状**：架构图中使用的术语与设计文档不一致

| 位置 | 当前术语 | 规范术语 | 影响 |
|------|---------|---------|------|
| 标题 | "交互架构图" | "Agent 实现流程" | 命名不够精准 |
| 列 1 | "用户 / 行业运营" | "用户输入（User Input）" | 角色混淆 |
| 列 3 步骤 2 | "会话字段继承" | "Field Memory 累积" | 未使用规范术语 |
| 底部模块 A | "对话历史（Turns）" | 应明确"最多 20 轮" | 缺少量化说明 |

**问题**：
- 架构图说"会话字段继承"，设计文档说"Field Memory 累积"
- 用户不知道"会话"="Session"
- 没有体现 Turn/Task Context/Field Memory 的层级关系

---

#### **问题 2：流程顺序与实际不符**

**现状**：第 3 列（LLM 编排层）的步骤顺序

```
当前顺序：
1. Schema 校验
2. 记忆系统
3. Skill 路由
4. 结果编排
```

**实际执行顺序**：
```
1. 记忆系统（先加载 Field Memory）
2. Schema 校验（检查累积字段 + 新字段）
3. Skill 路由（校验通过后）
4. 结果编排
```

**问题**：
- Schema 校验依赖 Field Memory（需要先合并字段）
- 架构图顺序误导了执行逻辑

---

#### **问题 3：缺少数据流向标注**

**现状**：只有箭头，没有数据流向说明

```
[第 2 列] ──→ [第 3 列]
   输出：task_type + 字段草稿 + 置信度
         ↓
      去哪了？
```

**缺失**：
- 字段草稿 → Field Memory（合并）
- task_type → Schema 校验（查必填字段）
- 校验结果 → Skill 路由 or 追问

---

#### **问题 4：记忆系统展示不够清晰**

**现状**：记忆系统只有一行说明

```
2. 记忆系统
   短期记忆：会话字段继承 + 任务状态管理
   长期记忆（未实现）：用户画像 + 历史任务
```

**问题**：
- 没有体现 Session → Turn → Task Context → Field Memory 的层级
- 没有展示数据流向（输入 → 累积 → 输出）
- 没有说明触发时机（每轮对话后）

---

#### **问题 5：底部辅助区域信息冗余**

**现状**：5 个模块，部分内容重复

```
A. 记忆系统
   - Session 内字段累积
   - 最近 20 轮对话历史
   - 任务状态管理
   - localStorage 持久化（24h）

B. Schema 校验
   - 必填字段校验
   - ✓ executable / ⚠ needs_clarification / ✗ blocked

C. Skill 路由
   - task_type → SKILL.md
   - 签名排查 / 回调排查 / 字段问答 / 人工闸门
```

**问题**：
- A 模块和第 3 列步骤 2 重复（都是记忆系统）
- B 模块和第 3 列步骤 1 重复（都是 Schema 校验）
- C 模块和第 3 列步骤 3 重复（都是 Skill 路由）

**原因**：
- 主流程图太简洁 → 补充到底部
- 但底部成了"重复说明区"而不是"扩展信息区"

## 🎯 优化建议

### **优化 1：术语规范化**

#### **修改点 1：标题与列名**

**Before**：
```
API 智能客服 Agent 交互架构图
列 1: 用户 / 行业运营
列 3: LLM 智能编排层 → 记忆系统 → 会话字段继承
```

**After**：
```
API 智能客服 Agent 实现流程
列 1: 用户输入（User Input）
列 3: LLM 智能编排层 → 记忆系统 → Field Memory 累积 + Task Context 管理
```

**理由**：
- "实现流程"比"交互架构"更准确（强调执行逻辑）
- "用户输入"比"用户/行业运营"更清晰（聚焦输入源）
- "Field Memory"是规范术语（与设计文档一致）

---

#### **修改点 2：底部模块术语**

**Before**：
```
A. 记忆系统
   • Session 内字段累积
   • 最近 20 轮对话历史
```

**After**：
```
A. 记忆系统（Short-term Memory）
   • Session（会话）：localStorage 持久化，24h TTL
   • Turn（对话轮次）：最近 20 轮，UI 显示 5 轮
   • Task Context（任务上下文）：当前任务状态快照
   • Field Memory（字段记忆）：Session 内累积，只增不减
```

**理由**：
- 明确 Session/Turn/Task Context/Field Memory 层级
- 补充关键参数（20 轮、24h TTL）
- 与设计文档完全对齐

---

### **优化 2：调整流程顺序**

#### **修改点：第 3 列步骤重排序**

**Before**：
```
3. LLM 智能编排层
   1. Schema 校验
   2. 记忆系统
   3. Skill 路由
   4. 结果编排
```

**After**：
```
3. LLM 智能编排层
   1. 记忆系统          ← 先加载 Field Memory
   2. Schema 校验       ← 检查累积字段 + 新字段
   3. Skill 路由        ← 校验通过后路由
   4. 结果编排
```

**数据流向标注**：
```
1. 记忆系统
   输入：新字段（第 2 列输出）
   处理：合并 Field Memory
   输出：完整字段集合

2. Schema 校验
   输入：完整字段集合
   处理：对比 requiredFields[task_type]
   输出：status（executable / needs_clarification / blocked）

3. Skill 路由
   输入：task_type + status（executable）
   处理：skills[task_type] → SKILL.md
   输出：Skill SOP 内容

4. 结果编排
   输入：Skill SOP + 文档 + 工具结果
   处理：LLM 生成回复
   输出：Agent 响应
```

**理由**：
- 符合实际代码执行顺序
- 明确数据流向（输入 → 处理 → 输出）
- 便于理解依赖关系

---

### **优化 3：增强记忆系统可视化**

#### **修改点：第 3 列步骤 1 扩展**

**Before（简洁版）**：
```
1. 记忆系统
   短期记忆：会话字段继承 + 任务状态管理
```

**After（分层展示）**：
```
1. 记忆系统
   ┌─ Session（会话）
   │  └─ Turn 1, Turn 2, ..., Turn N（最多 20 轮）
   │     ├─ Task Context（任务上下文）
   │     └─ Field Memory（字段记忆）
   └─ 数据流向：新字段 → 合并 → 输出完整字段集合
```

**方案 A：内联小图（推荐）**
```
1. 记忆系统
   Session → [Turn₁ Turn₂ ... Turn₂₀] → Field Memory 累积
   ↓ 每轮对话后自动保存到 localStorage（24h TTL）
```

**方案 B：保持简洁 + 底部扩展**
```
主流程：
1. 记忆系统
   加载 Field Memory + Task Context

底部辅助区域：
A. 记忆系统（详细）
   Session → Turn → Task Context → Field Memory
   [配层级关系图]
```

**推荐**：方案 B（保持主流程简洁）

---

### **优化 4：底部辅助区域重构**

#### **策略：从"重复说明"变为"扩展信息"**

**Before（重复型）**：
```
A. 记忆系统
   • Session 内字段累积        ← 与列 3 步骤 2 重复
   • 最近 20 轮对话历史

B. Schema 校验
   • 必填字段校验              ← 与列 3 步骤 1 重复

C. Skill 路由
   • task_type → SKILL.md      ← 与列 3 步骤 3 重复
```

**After（扩展型）**：
```
A. 记忆系统（架构细节）
   ┌─ Session 生命周期
   │  创建：页面打开 → sessionId 生成
   │  持久化：每轮对话后 saveSession()
   │  恢复：页面加载时 loadSession()
   │  清空：手动重置 or 超过 24h
   │
   ├─ Turn 数据结构
   │  { index, text, intent, status, missing, fields, timestamp }
   │
   ├─ Task Context 生命周期
   │  创建：首次识别任务
   │  更新：字段补充 / 状态变化
   │  覆盖：强烈新意图时
   │
   └─ Field Memory 策略
      只增不减（除非手动删除）
      跨轮累积（Turn 1 的 appid + Turn 2 的 error_code）

B. Schema 校验（必填字段明细）
   signature_debug   → api_path, appid, raw_sign_string, error_code, request_time
   callback_debug    → callback_url, order_id, trigger_action, request_time
   api_field_qa      → field_name
   credential_request → appid（触发人工审核）

C. Skill 路由（来源与维护）
   来源：历史 SOP 文档、专家经验、错误码手册
   格式：Markdown（适用场景 + 必要信息 + 排查 SOP + 常见错误）
   维护：发现新问题 → 编写 Skill → 添加路由映射

D. 知识与数据源（与列 2/3 交互）
   本地包 → 远程 llm-wiki → 向量检索 → LLM 上下文

E. 规则与风控（安全边界）
   appsecret 阻断 → blocked 状态 → 拒绝执行
```

**关键变化**：
- A 不重复"字段累积"，而是展示**生命周期 + 数据结构**
- B 不重复"校验逻辑"，而是展示**必填字段明细表**
- C 不重复"路由映射"，而是展示**来源与维护流程**

---

### **优化 5：增加数据流向可视化**

#### **修改点：在箭头上标注数据**

**Before（只有箭头）**：
```
[第 2 列] ──→ [第 3 列]
```

**After（标注数据）**：
```
[第 2 列]
  输出：task_type, fields_draft, confidence
    ↓
  ──────────────────────────────────→
    task_type="signature_debug"
    fields={appid:"xxx", error_code:"SIGN_INVALID"}
    confidence=0.86
  ──────────────────────────────────→
    ↓
[第 3 列 - 记忆系统]
  处理：合并 Field Memory
  输出：完整字段集合
    ↓
[第 3 列 - Schema 校验]
  处理：对比 requiredFields["signature_debug"]
  输出：status="needs_clarification", missing=["api_path", "raw_sign_string", "request_time"]
    ↓
[第 3 列 - 追问 or 路由]
  if status="needs_clarification" → 追问
  if status="executable" → Skill 路由
```

**实现方式**：
- 在箭头下方添加小字标注
- 使用不同颜色区分数据类型（蓝色=结构化字段，绿色=状态）

## 📐 具体修改方案

### **方案 A：最小改动（推荐）**

**改动量**：~30 行代码

**改动点**：
1. 列 3 步骤重排序（记忆系统 → Schema 校验 → Skill 路由 → 结果编排）
2. 术语规范化（会话字段继承 → Field Memory 累积）
3. 底部模块 A 扩展（增加生命周期说明）
4. 底部模块 B 扩展（增加必填字段明细表）

**效果**：
- ✅ 符合实际执行顺序
- ✅ 术语与设计文档一致
- ✅ 底部不再是"重复说明"
- ✅ 保持架构图简洁

---

### **方案 B：中等改动**

**改动量**：~80 行代码

**在方案 A 基础上增加**：
5. 数据流向标注（箭头下方小字）
6. 第 3 列步骤增加"输入 → 处理 → 输出"标注
7. 记忆系统增加小型层级图（Session → Turn → Field Memory）

**效果**：
- ✅ 数据流向清晰可见
- ✅ 便于理解依赖关系
- ⚠️ 架构图长度增加 ~20%

---

### **方案 C：大幅改动（不推荐）**

**改动量**：~200 行代码

**在方案 B 基础上增加**：
8. 拆分第 3 列为"编排层"和"执行层"（2 列变 3 列）
9. 增加"反馈循环"可视化（从输出回到记忆系统）
10. 增加"异常分支"流程图（blocked / needs_clarification）

**效果**：
- ✅ 完整的流程图
- ❌ 架构图过于复杂
- ❌ 阅读体验变差（违反"保持简洁"原则）

## 🎨 视觉优化建议

### **建议 1：配色优化**

**当前**：
- 蓝色（Schema 校验）
- 绿色（Skill 路由）
- 橙色（Skill 来源）

**建议增加**：
- **紫色**：记忆系统（与蓝色区分）
- **红色**：安全边界（blocked 状态）
- **灰色**：未实现功能（长期记忆）

**理由**：
- 不同颜色 = 不同模块类型
- 视觉上更容易区分

---

### **建议 2：图标优化**

**当前**：所有步骤都是圆形数字标签（1、2、3、4）

**建议**：
- 记忆系统：💾 存储图标
- Schema 校验：✓ 检查图标
- Skill 路由：🔀 路由图标
- 结果编排：📝 文档图标

**理由**：
- 图标比数字更直观
- 与底部辅助区域图标一致

---

### **建议 3：间距优化**

**当前**：第 3 列（LLM 编排层）和 Skills Panel 间距较紧

**建议**：
- 增加 20px 间距
- 用虚线箭头连接（Load / Invoke）

**理由**：
- 视觉上更清晰
- 强调"加载"和"调用"的区别
